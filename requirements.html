Software Requirements

This page defines the software requirements for the TRAVEO II Entry Family Starter Kit LED toggle feature. Each requirement is assigned a unique ID used for traceability between specification and implementation.

System Requirements

REQ-SYS-001 – Target Hardware

ID:

REQ-SYS-001

Type:

System

Priority:

Mandatory

Status:

Approved

Test Cases:

TC-001 – SW1 Pin Drive Mode After Port_Init, TC-002 – LED1 Pin Drive Mode After Port_Init

The software shall execute on the CYTVII-B-E-1M-SK evaluation board equipped with the CYT2B75XX microcontroller (32-bit Arm Cortex-M4F, QFP 100-pin, 0.5 mm pitch).

REQ-SYS-002 – Software Architecture

ID:

REQ-SYS-002

Type:

System

Priority:

Mandatory

Status:

Approved

Test Cases:

TC-012 – No Direct Register Access in SwcLedToggle

The software shall follow the AUTOSAR Classic layered architecture:

  • MCAL – Port Driver, DIO Driver

  • ECU Abstraction – I/O Hardware Abstraction (IoHwAb)

  • Application – Software Component (SWC)

Hardware Requirements

REQ-HW-001 – USER Switch Input

ID:

REQ-HW-001

Type:

Hardware

Priority:

Mandatory

Status:

Approved

Test Cases:

TC-001 – SW1 Pin Drive Mode After Port_Init, TC-007 – Dio_ReadChannel Returns STD_LOW When SW1 Pressed, TC-009 – IoHwAb Polarity Inversion for SW1

The USER Switch SW1 shall be connected to MCU port pin P7.0 (QFP pin 29), configured as a resistive pull-up digital input. The switch is active-LOW: logic LOW on the pin indicates a pressed state.

REQ-HW-002 – USER LED Output

ID:

REQ-HW-002

Type:

Hardware

Priority:

Mandatory

Status:

Approved

Test Cases:

TC-002 – LED1 Pin Drive Mode After Port_Init, TC-008 – Dio_WriteChannel Drives LED1 Active-LOW, TC-010 – IoHwAb Polarity Inversion for LED1

The USER LED LED1 (blue) shall be connected to MCU port pin P19.0 (QFP pin 77), configured as a strong-drive digital output. The LED is active-LOW: driving the pin LOW illuminates the LED.

Functional Requirements

REQ-FUNC-001 – LED Toggle on Button Press

ID:

REQ-FUNC-001

Type:

Functional

Priority:

Mandatory

Status:

Approved

Implements:

REQ-HW-001 – USER Switch Input, REQ-HW-002 – USER LED Output

Test Cases:

TC-004 – LED1 ON After First SW1 Press, TC-005 – LED1 OFF After Second SW1 Press

Pressing SW1 once shall turn LED1 ON. Pressing SW1 again shall turn LED1 OFF. Each confirmed press event shall toggle the LED state exactly once.

REQ-FUNC-002 – Falling-Edge Detection

ID:

REQ-FUNC-002

Type:

Functional

Priority:

Mandatory

Status:

Approved

Implements:

REQ-FUNC-001 – LED Toggle on Button Press

Test Cases:

TC-004 – LED1 ON After First SW1 Press, TC-005 – LED1 OFF After Second SW1 Press, TC-006 – No Repeated Toggle While Holding SW1

The software shall act on a single rising edge of the logical SW1 signal (INACTIVE to ACTIVE transition) per button press. Holding the button pressed shall not repeatedly toggle the LED.

Software Requirements

REQ-SW-001 – Port Initialisation

ID:

REQ-SW-001

Type:

Software

Priority:

Mandatory

Status:

Approved

Allocated To:

Port_Init() in src/Mcal/Port/Port.c

Test Cases:

TC-001 – SW1 Pin Drive Mode After Port_Init, TC-002 – LED1 Pin Drive Mode After Port_Init

The Port Driver shall initialise all configured pins before any application runnable is called. The initialisation order shall be:

  1. Set HSIOM to GPIO mode for each pin.

  2. Write the initial output value.

  3. Set the drive mode.

REQ-SW-002 – DIO Read Channel

ID:

REQ-SW-002

Type:

Software

Priority:

Mandatory

Status:

Approved

Allocated To:

Dio_ReadChannel() in src/Mcal/Dio/Dio.c

Test Cases:

TC-007 – Dio_ReadChannel Returns STD_LOW When SW1 Pressed

The DIO Driver shall read the current logical level of a channel from the hardware PRT_PS (Pin State) register and return STD_HIGH or STD_LOW.

REQ-SW-003 – DIO Write Channel

ID:

REQ-SW-003

Type:

Software

Priority:

Mandatory

Status:

Approved

Allocated To:

Dio_WriteChannel() in src/Mcal/Dio/Dio.c

Test Cases:

TC-008 – Dio_WriteChannel Drives LED1 Active-LOW

The DIO Driver shall write the requested logical level to the hardware PRT_DR (Data Register) of the specified channel.

REQ-SW-004 – IoHwAb Polarity Abstraction

ID:

REQ-SW-004

Type:

Software

Priority:

Mandatory

Status:

Approved

Allocated To:

IoHwAb.c in src/EcuAb/IoHwAb/

Test Cases:

TC-009 – IoHwAb Polarity Inversion for SW1, TC-010 – IoHwAb Polarity Inversion for LED1

The IoHwAb layer shall abstract the active-LOW hardware polarity of SW1 and LED1. The application shall use only logical values IOHWAB_SIG_ACTIVE / IOHWAB_SIG_INACTIVE without knowledge of physical pin polarity.

REQ-SW-005 – SWC Runnable Period

ID:

REQ-SW-005

Type:

Software

Priority:

Mandatory

Status:

Approved

Allocated To:

SwcLedToggle_Run10ms() in src/App/SwcLedToggle/SwcLedToggle.c

Test Cases:

TC-011 – SWC Runnable Called at 10 ms Period

The SwcLedToggle_Run10ms runnable shall be called with a period of 10 ms by the scheduler. This period provides implicit software debounce for the mechanical switch.

REQ-SW-006 – No Direct Register Access in Application Layer

ID:

REQ-SW-006

Type:

Software / Architecture

Priority:

Mandatory

Status:

Approved

Test Cases:

TC-012 – No Direct Register Access in SwcLedToggle

Application-layer code (SWC) shall not access hardware registers directly. All hardware interaction shall be routed through IoHwAb and MCAL.
